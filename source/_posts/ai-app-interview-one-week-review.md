## 3. 简历项目一：RestaurantOps Agent

### 3.1 项目一句话

RestaurantOps Agent 是一个面向餐饮运营场景的 AI Agent 应用，用 LangGraph 把菜单规划、预算校验、库存校验、采购单生成、人工审核和报告导出串成可追踪的多节点流程。

### 3.2 项目架构

核心模块：

- 前端演示：Streamlit
- 接口层：FastAPI
- 工作流编排：LangGraph StateGraph
- 工具层：Pydantic 类型化工具
- 数据层：SQLAlchemy + SQLite
- 测试：pytest
- 部署：Docker

核心流程：

1. 用户提交需求：人数、预算、偏好、时间等。
2. Agent 解析需求并初始化 state。
3. 菜品检索工具读取菜品库和供应商报价。
4. 菜单规划节点生成候选菜单。
5. 预算校验节点计算成本。
6. 库存校验节点检查食材是否足够。
7. 如果预算超标或库存不足，进入人工审核节点。
8. 审批通过后生成采购单草稿。
9. 将菜单计划、采购单、审批状态保存到数据库。
10. 生成 Markdown 报告并支持下载。

### 3.3 STAR 讲述模板

背景：餐饮运营中菜单设计、预算控制、库存检查、采购生成和人工审批通常是分散处理的，流程不透明，也容易出现人工遗漏。

任务：我希望用 Agent 工作流把这些环节串起来，做成一个可以演示、可以追踪、可以持久化的 AI 应用。

行动：我使用 LangGraph StateGraph 设计多节点流程，用 Pydantic 定义菜品检索、预算计算、库存检查、采购单生成等工具；使用 FastAPI 封装菜单生成、方案查询、方案审批和菜品库查询接口；用 SQLAlchemy 建模菜品、库存、供应商报价、菜单计划和采购单；最后用 Streamlit 做演示页面，并补充了 20 个 pytest 测试。

结果：项目最终支持从需求提交到菜单生成、预算/库存检查、人工审批、采购单生成和报告导出的完整闭环。通过这个项目，我对 Agent 的理解从“调用大模型”深入到了状态管理、工具调用、条件分支、业务数据注入和人工审核机制。

### 3.4 技术深挖回答模板

如果面试官问“你这个 Agent 具体怎么实现的”，可以这样回答：

> 我不是把所有逻辑都写在一个 prompt 里，而是用 LangGraph 把流程拆成多个节点。State 里会保存用户需求、候选菜品、预算结果、库存结果、采购单草稿、审批状态和错误信息。每个节点只负责一个明确任务，比如菜单规划、预算校验、库存校验。预算或库存不满足时，通过 conditional edge 进入人工审核节点。工具层用 Pydantic 定义输入输出，保证参数结构稳定。业务数据从 SQLite 读取，比如菜品、库存和供应商报价，这样生成结果不是纯模型编造，而是基于持久化数据。

### 3.5 高频问题

#### Q1：为什么用 LangGraph，而不是 LangChain 普通 Chain？

回答要点：

- 普通 Chain 更适合线性流程。
- 你的场景有状态流转、条件分支和人工审核。
- LangGraph 更像状态机，适合复杂 Agent 工作流。
- 可以清楚控制节点之间的流转，而不是完全交给模型自由决定。

参考回答：

> 因为我的流程不是简单的一问一答，而是有多个业务节点，比如菜单规划、预算校验、库存校验、采购单生成和人工审核。尤其预算超标和库存不足时，需要进入不同分支，所以我用 LangGraph 的 StateGraph 来建模。它可以把 Agent 流程拆成节点和边，每个节点处理 state 的一部分，条件边决定下一步走向，这比普通线性 Chain 更适合业务工作流。

#### Q2：你的 State 里有哪些字段？

建议字段：

- user_request：用户原始需求
- parsed_requirement：解析后的结构化需求
- candidate_dishes：候选菜品
- menu_plan：菜单方案
- budget_result：预算结果
- inventory_result：库存检查结果
- purchase_order：采购单草稿
- approval_status：审批状态
- report_markdown：最终报告
- errors：错误信息

参考回答：

> State 的作用是让各节点共享上下文。我会把用户需求、菜单方案、预算结果、库存结果、采购单草稿、审批状态和错误信息放在 state 里。比如预算校验节点读取菜单方案和供应商报价，输出 budget_result；库存校验节点读取菜单方案和库存快照，输出 inventory_result；后面的条件分支根据这两个结果决定是否进入人工审核。

#### Q3：human-in-the-loop 怎么实现？

回答要点：

- 定义触发条件：预算超标、库存不足、采购金额较高。
- 保存当前 state。
- 等待人工审批。
- 审批通过后继续生成采购单，拒绝则终止或重新规划。

参考回答：

> 我把 human-in-the-loop 设计成一个独立节点。当预算超标或库存不足时，条件边会把流程导向审核节点，而不是让 Agent 自己强行决定。系统会保存当前菜单方案、预算差额、库存缺口和采购建议，前端展示给人工审核。审批通过后更新 approval_status，再继续生成采购单；如果拒绝，就终止或重新规划。

#### Q4：Pydantic 工具有什么作用？

回答要点：

- 定义工具输入输出 schema。
- 做参数校验。
- 避免 LLM 输出格式混乱。
- 方便测试和维护。

参考回答：

> Pydantic 主要用于定义结构化工具的输入输出。例如预算工具需要菜单 ID、菜品数量、供应商报价等参数，库存工具需要食材需求和库存快照。用 Pydantic 后，输入字段、类型和必填项都比较明确，减少模型或接口传参不稳定的问题，也方便写单元测试。

#### Q5：如果工具调用失败怎么办？

回答要点：

- 捕获异常。
- 写入 state.errors。
- 返回可解释错误。
- 必要时重试或进入人工处理。

参考回答：

> 工具失败不能让整个流程静默失败。我会在工具层捕获异常，比如数据库查询失败、库存数据缺失、参数校验失败，然后把错误写入 state.errors，同时返回明确的错误码或错误信息。如果是临时错误可以重试，如果是业务数据缺失，就转人工处理或提示补充数据。

## 4. 简历项目二：多模态 RAG 知识库问答系统

### 4.1 项目一句话

多模态 RAG 知识库问答系统支持 PDF、Markdown、图片等知识源，通过文档解析、OCR/VLM、Embedding、向量检索、BM25、上下文构造和 RAGAS 评估，实现个人知识场景下的问答增强。

### 4.2 RAG 基础流程

RAG 全称 Retrieval-Augmented Generation，即检索增强生成。

离线阶段：

1. 文档加载：读取 PDF、Markdown、图片等。
2. 文档解析：提取文本、表格、图片、公式。
3. 文本切分：把长文档切成 chunk。
4. 向量化：用 Embedding 模型把 chunk 转成向量。
5. 入库：存入 Chroma 等向量数据库。

在线阶段：

1. 用户提问。
2. 对问题向量化。
3. 检索相关 chunk。
4. 可选：BM25 混合检索或 rerank。
5. 构造 prompt 上下文。
6. LLM 基于上下文生成答案。
7. 返回答案和引用来源。

### 4.3 STAR 讲述模板

背景：个人知识库里有 PDF、Markdown、图片等多种资料，直接让大模型回答容易受上下文长度限制，也容易产生幻觉。

任务：我希望做一个多模态 RAG 系统，支持常见知识源的解析、索引、检索和问答，并对效果和部署问题进行调试。

行动：我配置了 LLM 函数、视觉模型函数和 Embedding 函数，使用 GPT-4o-mini 做文本生成，GPT-4o 做图像理解，text-embedding-3-large 做向量化；分析了解析器注册、解析缓存、文档状态管理、多模态状态缓存、图片/表格/公式处理器；结合测试用例验证 PaddleOCR 接入、解析参数透传、内容 schema、页码保留和自定义 parser registry；还定位了 tiktoken 离线缓存下载超时和 MinerU 本地服务健康检查失败等部署问题。

结果：最终完成本地集成调试，对多模态 RAG 从解析、索引、检索、生成、评估到离线部署的链路有了完整理解。

### 4.4 高频问题

#### Q1：RAG 解决什么问题？

回答要点：

- 解决大模型知识过期问题。
- 解决私有知识无法直接进入模型的问题。
- 降低幻觉。
- 提供可追溯来源。

参考回答：

> RAG 主要解决大模型无法直接掌握私有知识、知识可能过期、回答容易幻觉的问题。它先从外部知识库检索相关内容，再把这些内容作为上下文交给大模型生成答案。这样回答会更贴近业务文档，也更容易提供引用来源。

#### Q2：chunk size 怎么选？

回答要点：

- 太小：语义不完整，召回碎片化。
- 太大：噪声多，占用上下文。
- 要根据文档结构、段落、标题、表格边界选择。
- 可以通过召回效果和 RAGAS 指标调优。

参考回答：

> chunk size 不能固定看数字，要看文档结构。太小容易丢失上下文，太大又会引入噪声。我一般会优先按标题、段落和语义边界切分，再结合 overlap 保留上下文连续性。如果发现召回内容不完整，就适当增大 chunk 或增加 overlap；如果回答混入无关信息，就减小 chunk 或加 rerank。

#### Q3：BM25 和向量检索有什么区别？

基础原理：

- BM25 是基于关键词匹配的稀疏检索算法。
- 向量检索是基于语义相似度的稠密检索方法。

BM25 优点：

- 对专有名词、编号、精确关键词效果好。
- 不依赖 Embedding 模型。
- 可解释性较强。

向量检索优点：

- 能处理语义相似表达。
- 用户问题和文档措辞不一致时仍可能召回。

参考回答：

> BM25 更偏关键词匹配，比如产品编号、人名、接口名、专业术语；向量检索更偏语义匹配，比如“怎么报销”和“费用报销流程”虽然字面不同，但语义接近。实际 RAG 中常用混合检索，因为 BM25 能补精确匹配，向量检索能补语义召回。

#### Q4：RAGAS 评估什么？

常见指标：

- faithfulness：答案是否忠实于上下文。
- answer relevancy：答案是否回答了问题。
- context precision：检索到的上下文中有多少是有用的。
- context recall：需要的信息是否被检索出来。

参考回答：

> RAGAS 用来评估 RAG 的回答质量和检索质量。比如 faithfulness 看答案有没有根据上下文胡编，answer relevancy 看答案是否切题，context precision 看检索内容是否有效，context recall 看关键证据是否被召回。它适合用来对比不同切分、检索和 prompt 策略的效果。

#### Q5：多模态 RAG 中图片怎么处理？

回答要点：

- OCR：提取图片中的文字。
- VLM：理解图片内容、图表、截图。
- 图片路径安全校验。
- base64 编码传给视觉模型。
- 将图像描述或 OCR 文本进入索引。

参考回答：

> 图片可以分两类处理。如果图片主要是文字，比如扫描件或截图，可以先用 OCR 提取文本，再进入普通 RAG 流程。如果图片包含图表、结构或视觉语义，则可以用 VLM 生成描述。工程上还要做图片路径安全校验，避免任意文件读取，然后将图片转 base64 输入视觉模型。

## 5. OpenAI API 与大模型应用基础

### 5.1 System Prompt

基础原理：

System prompt 用来定义模型角色、行为边界、输出格式和禁止事项。它的优先级通常高于用户输入。

高频考点：

- system prompt 和 user prompt 的区别是什么？
- 如何让模型稳定输出 JSON？
- 如何减少幻觉？

参考回答：

> system prompt 更像系统规则，用来告诉模型角色、任务边界和输出要求；user prompt 是用户本轮具体问题。比如在我的菜单生成项目里，system prompt 会规定模型必须基于菜品库和库存数据生成菜单，不能编造不存在的菜品；user prompt 才是用户输入的预算、人数和偏好。

### 5.2 Few-shot

基础原理：

Few-shot 是在 prompt 中给模型几个示例，让模型模仿示例的格式和推理方式。

高频考点：

- zero-shot 和 few-shot 区别？
- 什么时候需要 few-shot？
- 示例如何选择？

参考回答：

> zero-shot 是不给示例直接让模型完成任务，few-shot 是提供几个输入输出样例。对于格式要求强、业务规则复杂的任务，我会使用 few-shot，比如给几个菜单规划示例，让模型学会预算字段、菜品结构和报告格式。

### 5.3 结构化输出

基础原理：

结构化输出是让模型按照 JSON、schema 或 Pydantic 结构返回结果，便于程序解析。

高频考点：

- 为什么不能只让模型输出自然语言？
- JSON 输出失败怎么办？
- 结构化输出和函数调用有什么关系？

参考回答：

> AI 应用不能只依赖自然语言，因为后端需要解析字段，比如菜单名称、菜品列表、预算金额、库存缺口等。结构化输出可以让模型结果直接进入业务流程。如果 JSON 解析失败，可以用 schema 约束、重试、修复解析器或把关键逻辑交给工具函数计算。

### 5.4 Function Calling / Tool Calling

基础原理：

Tool calling 是让模型决定是否调用外部工具，并生成符合 schema 的工具参数。真正的工具执行发生在程序侧。

高频考点：

- 工具调用和普通 prompt 有什么区别？
- 模型会不会真的执行工具？
- 如何保证工具安全？

参考回答：

> 工具调用的重点是让模型选择工具并生成参数，真正执行工具的是后端程序。比如模型可以决定调用库存检查工具，并给出菜品和数量，但数据库查询、库存计算和结果返回都是程序执行。安全上要限制工具权限、校验参数、记录日志，不能让模型随意执行任意代码。

### 5.5 成本控制

常见策略：

- 小模型优先，大模型兜底。
- 缓存重复问题和解析结果。
- 控制上下文长度。
- RAG 只传最相关内容。
- 图片理解按需调用，不要每次都调用 VLM。
- 对结构化任务尽量用规则或工具计算。

参考回答：

> 成本控制可以从模型选择、上下文长度和调用次数三方面做。比如普通文本生成用 GPT-4o-mini，图片理解再用 GPT-4o；RAG 只传 top-k 相关 chunk；OCR 和解析结果做缓存；预算和库存计算不交给模型，而是用工具函数完成。

### 6.1 FastAPI 是什么

## 8. SQLAlchemy 与 ORM 基础

### 8.1 ORM 是什么

基础原理：

事务保证一组数据库操作要么全部成功，要么全部失败。

ACID：

- Atomicity：原子性。
- Consistency：一致性。
- Isolation：隔离性。
- Durability：持久性。

高频考点：

### 8.4 索引

基础原理：

索引是数据库为了加快查询建立的数据结构，常见是 B+ 树。

高频考点：

- 什么字段适合建索引？
- 索引有什么缺点？

参考回答：

> 索引类似书的目录，可以减少全表扫描。适合在经常用于查询条件、排序、关联的字段上建索引，比如 menu_plan_id、status、created_at。缺点是会占空间，写入和更新时也要维护索引，所以不是越多越好。

## 10. Redis 基础

你的简历写了熟悉 Redis 的常见使用场景，了解缓存、持久化、主从复制、哨兵、分片集群。面试时不要说得太深，但要能回答基础问题。

### 10.1 Redis 是什么

Redis 是基于内存的 key-value 数据库，读写速度快，常用于缓存、分布式锁、计数器、排行榜、会话存储、限流等场景。

简历关联：

> 在 AI 应用中，Redis 可以用来缓存 RAG 检索结果、模型回答、用户会话状态、任务状态，也可以做接口限流和分布式锁。

### 10.2 高频考点 1：缓存

基础原理：

缓存是把热点数据放在 Redis 中，减少数据库查询压力，提高响应速度。

常见问题：

- 缓存穿透：查询不存在的数据，每次都打到数据库。
- 缓存击穿：热点 key 过期，大量请求同时打到数据库。
- 缓存雪崩：大量 key 同时过期，数据库压力激增。

参考回答：

> Redis 最常见用途是缓存热点数据。比如 AI 应用中，某些文档解析结果、Embedding 结果或重复问答结果可以缓存，避免每次都重新计算。缓存问题上，穿透可以用布隆过滤器或缓存空值，击穿可以用互斥锁，雪崩可以给过期时间加随机值。

### 10.3 高频考点 2：持久化

基础原理：

Redis 是内存数据库，但支持把数据持久化到磁盘。

两种方式：

- RDB：定期快照，恢复快，但可能丢失最近数据。
- AOF：记录写命令，数据更完整，但文件更大。

参考回答：

> RDB 像定期拍快照，性能好、恢复快，但宕机时可能丢失最近一次快照后的数据。AOF 是追加写命令，数据安全性更高，但文件更大、恢复可能更慢。生产中可以结合使用。

### 10.4 高频考点 3：主从复制、哨兵、集群

基础原理：

- 主从复制：主节点负责写，从节点复制数据并承担读。
- 哨兵：监控主节点，故障时自动选主。
- 集群：通过分片把数据分布到多个节点。

参考回答：

> 主从复制解决读扩展和备份问题，哨兵解决主节点故障自动切换问题，集群解决单机容量和吞吐瓶颈问题。对于初级岗位，我理解它们的使用场景即可，真正生产配置还需要结合业务和运维经验。

### 10.5 Redis 面试回答边界

如果被问“你项目里用 Redis 了吗？”

推荐回答：

> 我当前简历里的 Agent 和 RAG 项目主要使用 SQLite 和本地缓存，没有把 Redis 作为核心依赖。但我理解 Redis 在这类项目里的典型用法，比如缓存文档解析结果、Embedding、RAG 检索结果、模型回答，以及做接口限流和任务状态缓存。如果项目上线，我会优先考虑把高成本、重复计算的数据放到 Redis。

## 11. Kafka 基础

你的简历写了了解 Kafka 的异步解耦、削峰填谷和可靠性保障机制。面试时需要能讲清楚“它是什么、为什么用、怎么保证可靠”。

### 11.1 Kafka 是什么

Kafka 是分布式消息队列 / 流式数据平台，用于异步通信、系统解耦、削峰填谷和日志数据处理。

简历关联：

> 在 AI 应用中，Kafka 可以用于异步处理文档解析、Embedding 生成、批量索引、报告生成、大模型调用任务等耗时操作。

### 11.2 高频考点 1：为什么用消息队列

核心作用：

- 异步处理：请求先入队，后台慢慢处理。
- 解耦：生产者不需要直接依赖消费者。
- 削峰填谷：高峰请求先堆在队列里。
- 提高可靠性：消息可持久化、可重试。

参考回答：

> 消息队列适合处理耗时任务。比如用户上传大量 PDF 后，不一定要同步等待解析、切分、Embedding 和入库全部完成，可以先把任务写入 Kafka，后台消费者异步处理。这样接口响应更快，系统之间也更解耦。

### 11.3 高频考点 2：Topic、Partition、Consumer Group

基础概念：

- Topic：消息主题。
- Partition：分区，提高并行度。
- Producer：生产者，发送消息。
- Consumer：消费者，读取消息。
- Consumer Group：消费者组，同组内分摊消费。

参考回答：

> Kafka 里消息写入 Topic，Topic 可以分多个 Partition 来提高并发。消费者通过 Consumer Group 消费消息，同一个组里的多个消费者会分摊不同分区。这样既能保证扩展性，又能避免同一条消息被同组多个消费者重复处理。

### 11.4 高频考点 3：可靠性保障

常见机制：

- 副本机制：Partition 有多个副本。
- ACK：生产者等待 broker 确认。
- Offset：消费者记录消费位置。
- 重试：失败后重新消费。

参考回答：

> Kafka 的可靠性主要靠副本、ACK 和 offset。生产者发送消息时可以设置 ack 确认级别，broker 通过副本保证节点故障时数据不丢，消费者通过 offset 记录消费进度。如果消费失败，可以不提交 offset，让消息后续重新消费。

### 11.5 Kafka 面试回答边界

如果被问“你实际用过 Kafka 吗？”

推荐回答：

> 我目前没有在简历项目里把 Kafka 作为核心组件落地，更多是理解它在 AI 应用里的使用场景。比如文档批处理、Embedding 生成、异步报告生成都适合用消息队列。我的理解重点是异步解耦、削峰填谷、Topic/Partition/Consumer Group 和 offset 可靠消费机制。

## 14. HTTP / HTTPS / TCP/IP 基础

### 14.1 HTTP 是什么

HTTP 是应用层协议，用于客户端和服务器之间传输请求和响应。

高频考点：

- GET 和 POST 区别？
- HTTP 状态码？
- HTTP 和 HTTPS 区别？

参考回答：

> HTTP 是明文传输，HTTPS 在 HTTP 基础上加入 TLS 加密，能保证传输加密、身份认证和数据完整性。GET 通常用于查询，参数可以放 URL；POST 通常用于提交数据或创建资源，请求体可以承载更复杂的数据。

### 14.2 常见状态码

- 200：请求成功。
- 201：创建成功。
- 400：请求参数错误。
- 401：未认证。
- 403：无权限。
- 404：资源不存在。
- 422：参数校验失败。
- 500：服务器内部错误。

### 14.3 TCP 三次握手

基础原理：

TCP 建立连接需要三次握手：

1. 客户端发送 SYN。
2. 服务端返回 SYN + ACK。
3. 客户端返回 ACK。

高频考点：

- 为什么要三次握手？
- TCP 和 UDP 区别？

参考回答：

> 三次握手是为了确认双方的发送和接收能力都正常。TCP 是面向连接、可靠、有序的传输协议；UDP 无连接、不保证可靠，但延迟低，适合实时音视频等场景。

## 15. 操作系统基础

### 15.1 进程和线程

基础原理：

- 进程是资源分配的基本单位。
- 线程是 CPU 调度的基本单位。
- 一个进程可以包含多个线程。

高频考点：

- 进程和线程区别？
- 多线程为什么有安全问题？
- Python GIL 是什么？

参考回答：

> 进程拥有独立地址空间，线程共享进程内存，所以线程间通信更方便，但也更容易出现并发安全问题。Python 中 CPython 有 GIL，同一时刻通常只有一个线程执行 Python 字节码，所以 CPU 密集型任务多线程不一定提升明显，IO 密集型任务仍然适合多线程或异步。

### 15.2 死锁

基础原理：

死锁是多个线程或进程互相等待对方释放资源，导致都无法继续执行。

四个条件：

- 互斥。
- 占有且等待。
- 不可抢占。
- 循环等待。

参考回答：

> 死锁可以通过破坏四个条件之一解决，比如按固定顺序加锁避免循环等待，设置锁超时，或者尽量减少锁粒度。

## 16. 数据结构与算法基础

### 16.1 常见数据结构

必须能简单解释：

- 数组：连续内存，随机访问快，插入删除慢。
- 链表：插入删除方便，随机访问慢。
- 栈：先进后出。
- 队列：先进先出。
- 哈希表：通过哈希函数快速查找。
- 树：层级结构。
- 堆：常用于优先队列。
- 图：节点和边，适合建模关系。

### 16.2 哈希表

高频考点：

- 哈希冲突是什么？
- 如何解决哈希冲突？
- 字典为什么查询快？

参考回答：

> 哈希表通过哈希函数把 key 映射到数组下标，所以平均查询复杂度接近 O(1)。哈希冲突是不同 key 映射到同一位置，常见解决方式有链地址法和开放寻址法。

### 16.3 图和 Agent 工作流的关联

参考回答：

> LangGraph 的工作流可以类比成图结构，节点是任务步骤，边是执行流转，条件边根据 state 决定下一步走向。我的 Agent 项目本质上就是把业务流程建模成有向图。

### 17.1 PyTorch 是什么

PyTorch 是深度学习框架，常用于模型训练、推理和实验。

高频考点：

- Tensor 是什么？
- 训练和推理区别？

### 17.2 YOLO

基础原理：

高频考点：

### 17.3 U-Net

基础原理：

U-Net 是语义分割模型，常用于医学图像分割。它有编码器和解码器结构，并通过跳跃连接保留细节。

高频考点：

- 语义分割和目标检测区别？
- U-Net 为什么适合医学图像？

参考回答：

> 语义分割是给每个像素分类，目标检测是给目标画框。U-Net 适合医学图像，是因为它能在小样本场景下通过编码器提取语义，通过解码器恢复空间细节，跳跃连接可以保留边缘信息。

## 21. 高频基础问答速记

### 21.1 数据库

Q：事务是什么？  
A：事务是一组操作，要么全部成功，要么全部失败，用来保证数据一致性。

Q：索引是什么？  
A：索引是加速查询的数据结构，常见是 B+ 树，但会增加写入维护成本。

Q：什么字段适合建索引？  
A：经常作为查询条件、排序字段、关联字段，且区分度较高的字段。

Q：慢 SQL 怎么优化？  
A：看 explain、加索引、减少返回字段、优化 join、避免函数导致索引失效、合理分页。

### 21.2 Redis

Q：Redis 为什么快？  
A：主要因为基于内存、数据结构高效、单线程避免锁竞争、IO 多路复用。

Q：缓存穿透怎么解决？  
A：缓存空值、布隆过滤器、参数校验。

Q：缓存雪崩怎么解决？  
A：过期时间加随机值、热点数据不过期、多级缓存、限流降级。

### 21.3 Kafka

Q：Kafka 为什么能削峰？  
A：请求高峰时消息先写入队列，消费者按能力慢慢处理。

Q：Partition 有什么作用？  
A：提高并行度和吞吐量。

Q：Offset 是什么？  
A：消费者消费到某个位置的记录，用于恢复和重试。

### 21.4 HTTP

Q：GET 和 POST 区别？  
A：GET 用于查询，参数通常在 URL；POST 用于提交或创建，请求体承载数据。

Q：HTTPS 为什么安全？  
A：通过 TLS 加密传输，并提供身份认证和完整性校验。

Q：状态码 401 和 403 区别？  
A：401 是未认证，403 是已认证但无权限。

### 21.5 操作系统

Q：进程和线程区别？  
A：进程是资源分配单位，线程是调度单位；线程共享进程资源。

Q：死锁条件？  
A：互斥、占有且等待、不可抢占、循环等待。

Q：IO 密集和 CPU 密集区别？  
A：IO 密集主要等待网络或磁盘，CPU 密集主要消耗计算资源。
